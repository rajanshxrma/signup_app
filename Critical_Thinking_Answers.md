Part 1 — Activity 09

1) It uses StatefulWidget because the page has stuff that changes — like when someone types in the text fields or when validation errors pop up. StatelessWidget can't handle any of that because it doesn't keep track of changes. If you tried using StatelessWidget here, the form basically wouldn't work. Like if you hit "Sign Up" with empty fields, the red error messages wouldn't show up because a StatelessWidget can't rebuild itself to display new stuff. The text controllers also wouldn't really work right since there's no state to hold onto.

2) The _formKey is basically how we talk to the Form widget from somewhere else in the code. When the user presses the Sign Up button, we need a way to tell the form "go check if everything is filled in right." That's what _formKey.currentState!.validate() does. It's "global" because it needs to be unique in the whole app — Flutter uses it to find the exact Form widget it belongs to even when the screen rebuilds. Without it we'd have no way to trigger validation on all the fields at once from the button press.

3) So the widget tree is basically like a family where each member has their own job. MyApp is the great-grandparent, it just kicks everything off with runApp() and doesn't really do anything else. MaterialApp is like the grandparent who sets the rules — it decides the theme colors and how navigation works for everyone below it. Scaffold is the parent who gives structure — it provides the app bar at the top and the body area where content goes. Form and TextFormField are the kids doing all the actual work — handling input and validation. This shows encapsulation because each widget only worries about its own thing. Scaffold doesn't know anything about form validation, and Form doesn't care about what color the theme is. Separation of concerns basically — everything has one job.

4) The current check just looks for an "@" symbol which is pretty weak. Something like "@" by itself or "hello@" or "@@@" would all pass as "valid" which obviously aren't real emails. A better version would use a regex that actually checks that there's something before the @, a domain after it, and a proper ending like .com or .edu. Something like RegExp(r'^[\w\-\.]+@([\w\-]+\.)+[\w\-]{2,4}$') would work much better.

5) SnackBar is kinda limited because it disappears after a few seconds so someone might not even notice it. It's also small and sits at the bottom of the screen. Plus it doesn't stop the user from doing other stuff, which might not be what you want after a signup. Better options would be an AlertDialog popup that the user has to tap to dismiss, just navigating to a new welcome page (which is what I did for the bonus) so the user clearly sees they signed up, or showing a success message right on the page itself where the form used to be.

6) When the widget gets removed (like if you navigate to another page), the controllers are supposed to be cleaned up by calling .dispose(). If you forget to do that, the controllers and their listeners just keep sitting in memory doing nothing. That's a memory leak. In a small app you probably wouldn't notice, but if you have a bunch of screens and the user keeps navigating around, all those undisposed controllers add up and the app starts using way more memory than it should. Could eventually make things slow or crash.

7) It's like a stack of cards. Whatever's on top is what you see. When you're on the signup page the stack just has SignupPage. After calling Navigator.push() to go to the welcome screen the stack has SignupPage at the bottom and WelcomeScreen on top. SignupPage is still there underneath, just hidden. When you hit back or call Navigator.pop(), WelcomeScreen gets removed and you're back to seeing SignupPage again.

8) obscureText just hides the dots on screen, that's it. For a real app you'd want way more. On the app side you'd make passwords require a mix of uppercase, lowercase, numbers, and special characters, show a strength meter so users pick better passwords, and don't store the password anywhere on the device in plain text. On the server side you'd hash passwords with something like bcrypt before saving them, always use HTTPS so the password isn't sent as plain text over the network, lock accounts after too many failed login attempts, and add two-factor authentication.

9) In Flutter, you write the form structure, the validation, and the styling all in Dart. Everything is in one place which is nice. The validator callback is right there on each field so you can see exactly what rules each field has. With HTML forms on the web, you have HTML for structure, CSS for styling, and JavaScript for validation logic — three different things to manage. HTML5 does have some built-in stuff like required and type="email" but for anything custom you need JS. Flutter's advantage is everything being in one language and the validator being right next to the field. The downside is there's a bigger learning curve compared to just writing an HTML form.

10) First, dispose the controllers — the original code never calls .dispose() on any of the TextEditingControllers which leaks memory every time the widget is destroyed. I added the dispose() override in my version to fix this. Second, add SingleChildScrollView — the original just uses a Column which means on smaller phones or when the keyboard pops up you get that "bottom overflowed by X pixels" error. Wrapping it in a scroll view fixes that. Third, pull out hardcoded strings — stuff like "Please enter your name" and the minimum password length of 6 are just hardcoded everywhere. It'd be cleaner to put those in constants at the top of the file so they're easier to find and change later.

Part 2 — Activity 10

1) Starting over would mean rewriting all the validation logic, the controllers, the form key setup — basically everything we already got working. By extending Part 1, we know the core form already validates correctly so we can just focus on the new stuff like adding screens and animations. It also means less chance of introducing new bugs since the base code was already tested. Plus in a real job you'd never throw away working code just to restructure it, you'd refactor it.

2) It made it way easier honestly. When everything was in one big main.dart, if two people tried to edit it at the same time you'd get merge conflicts constantly. With separate files — welcome_screen.dart, signup_screen.dart, success_screen.dart — each person could work on their own screen without stepping on anyone else's code. The only tricky part was making sure we all agreed on the same color scheme and naming so things looked consistent when we put it all together.

3) If you just push the success screen on top, the signup form is still sitting in the navigation stack underneath. That means the user could hit the back button and land on the filled-out form again, and potentially resubmit it. Using pushReplacement removes the signup screen from the stack entirely so there's no way to go back and accidentally create a duplicate account. It's a cleaner flow — once you've signed up, that step is done.

4) The confetti on the success screen honestly made the app feel way more polished. Before adding it, the success page was just a green checkmark and some text which felt pretty flat. The confetti plus the scale-in animation on the checkmark made it feel like the app was actually celebrating with you. The loading spinner on the signup button was also a nice touch because it made the transition feel intentional instead of just instantly jumping to the next screen.
